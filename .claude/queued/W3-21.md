# W3-21 — Rewrite the default report-delivery email body

Ford pinned this overnight (Note2): the current default delivery email body is "terrible." He sketched the replacement:

```
Dear {client_name},

Here is your {report_type} from {period_start} to {period_end}. Please reach out with any questions.

Thank you,
{operator_name}
{operator_email}
```

## What to change

1. **Email body** (in `api/email_templates.py` or wherever the default delivery template lives — check `api/notify.py::send_workbook_email` and `api/delivery.py`): replace the existing default body with the version above. Use the project's existing merge-tag syntax (`{{client_name}}` etc. — match what the EmailCustomization card supports).
2. **Subject line** should be operator-tone too: `"{report_type} for {client_name} — {period_start} to {period_end}"` (no Solar Operator branding in subject).
3. **From / Reply-To**:
   - If the tenant has a verified custom-From domain (the path you built/will-build in W3-12), send From = operator's address.
   - Otherwise send From = `admin@solaroperator.org` (platform fallback — required for Resend deliverability) but ALWAYS set `Reply-To: <operator_email>` so the client's reply goes to the operator, not us.
4. **Merge tags to introduce** (if not already present): `{{report_type}}`, `{{period_start}}`, `{{period_end}}`, `{{operator_name}}`, `{{operator_email}}`. `{{client_name}}` should already exist. Wire these in `api/delivery.py` where the template is rendered — `report_type` = "GMCS quarterly report" (or whatever the writer name produces), the period dates come from the workbook's reporting window (the writer knows it — pass it through).
5. **EmailCustomization card on /account.html**: update the placeholder / default-preview text to show the new body. The user can still override, but the defaults are now sane.
6. **Don't break existing customized tenants.** If `tenant.report_email_body` is set, leave it alone — only the DEFAULT changes. New tenants and tenants who haven't customized get the new body.
