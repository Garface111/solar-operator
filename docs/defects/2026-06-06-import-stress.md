## Defects Fixed Inline

### defect-import-01: No text normalisation at the ingest boundary
- **Test:** tests/test_import_stress.py::TestCleanText (8 tests), TestNameCollisionWeirdness::test_encoding_cleaned_*, TestDomainLogic::test_gmcs_shape_with_encoding_in_a1
- **Repro:** Upload a spreadsheet whose array-name cells contain smart quotes (U+2018/2019/201C/201D), NBSP (U+00A0), em dash (U+2014), zero-width space (U+200B), BOM (U+FEFF), in-cell CRLF, or Python byte-string repr (b'name'). The raw character survives openpyxl `str(cell_value)` unchanged.
- **Root cause:** `_normalize()` used `str(v).strip()` only. No Unicode normalisation was applied at any point in the three ingest paths (GMCS extractor, LLM flatten, heuristic parser).
- **Impact:** Smart-quote array names land in the DB as e.g. `'Chester'` (curly) instead of `'Chester'` (straight). The nepool_map / fuzzy-match lookups compare DB strings to future imports, so a second upload of the same roster fails to detect the collision and creates duplicate Array records.
- **Fix:** Added `clean_text(s)` public helper in `api/ingest.py`. Updated `_normalize()` to call `clean_text()` on every string-typed field (operator_name, array_name, gmp_account_number, notes). NEPOOL field is handled separately by `_coerce_nepool()`.
- **Files touched:** `api/ingest.py` (new `clean_text` function, updated `_normalize`)

### defect-import-02: NEPOOL GIS ID not coerced or validated
- **Test:** tests/test_import_stress.py::TestNepoolCoercion (16 tests)
- **Repro:** Upload a spreadsheet with NEPOOL values like `53984.0` (Excel float), `"53,984"` (thousands sep), `"#REF!"` (formula error), `"=SUM(A1:A5)"` (formula text), or `"53984; 53985"` (two IDs in one cell). All were stored verbatim in `Array.nepool_gis_id`.
- **Root cause:** `_normalize()` used `str(v).strip()` for every field including nepool_gis_id, with no type-checking or format validation. The DB column is an untyped string; no constraint caught the garbage.
- **Impact (SEV-5 risk):** `nepool_gis_id = "53984.0"` does not match `nepool_map["53984"]` in the preview collision-detection loop. An operator who imports the same array twice (once from a GMCS export, once from a NEPOOL portal CSV) would see no collision warning and unknowingly create a duplicate Array record. Bill attribution then splits kWh between two records for the same physical array.
- **Fix:** Added `_coerce_nepool(raw)` in `api/ingest.py` that: strips Excel apostrophe and whitespace, accepts float-with-zero-frac and scientific notation, strips comma thousands-separators, maps sentinel strings (N/A, TBD, -, NONE) to None without error, rejects formula errors (#…) and formula text (=…) with a structured error message, validates the result as 4–6 digits. Updated `_normalize()` to call it and attach a `nepool_parse_error` key to the row when coercion fails — row is KEPT in preview so the operator can correct it, not silently dropped. Also fixed a bug uncovered during test iteration: the sentinel check used `s.upper()` against a lowercase sentinel set, causing "N/A" and "TBD" to not be recognised as sentinels and to fall through to the digit check, producing spurious error messages.
- **Files touched:** `api/ingest.py` (new `_coerce_nepool`, `_NEPOOL_SENTINELS`, `_NEPOOL_RE`, updated `_normalize`)

## Defects Open (xfail strict)

### defect-import-03: Banner/title row at top breaks heuristic column parser
- **Test:** tests/test_import_stress.py::TestSheetStructureChaos::test_banner_row_at_top_breaks_heuristic (xfail)
- **Severity:** medium
- **Repro:** Upload an xlsx whose first visible row is a title banner (e.g. "Solar Array Portfolio Q3 2024") with no column-keyword matches. The heuristic parser treats it as the header row, sets `col_field = {}` (no matches), and falls into positional mode for all subsequent rows. The real header row (row 2) is treated as data and emits a spurious record whose `array_name` = "Array Name" and `operator_name` = "Client". The actual first data row is correctly parsed. Result: N+1 rows instead of N, with one garbage row containing column-header text as field values.
- **Repro detail:** Affects `.xlsx` files where `_llm_extract_hierarchical` returns None (no API key configured). With an LLM key, the LLM typically handles banners correctly.
- **Recommended fix:** Scan the first 5 rows and pick the one with the most keyword matches as the header, rather than always using row 0. A score of 2+ matching keywords (out of 5 canonical fields) is a reliable header signal. Alternatively, skip rows where none of the tokens match any keyword and the row has only 1 non-empty cell (typical of a title banner).
- **Estimated effort:** S

### defect-import-04: Merged header cells cause NEPOOL column-index shift
- **Test:** tests/test_import_stress.py::TestSheetStructureChaos::test_merged_header_cells_cause_column_shift (xfail)
- **Severity:** medium
- **Repro:** Create an xlsx where the header row has "Array Name" merged across columns A:B. openpyxl read_only mode returns the value for A only; B returns None → "" in `_xlsx_to_text`. The heuristic parser maps col 0 → array_name, col 2 → nepool_gis_id. Data rows have no merge: col 0 = array name, col 1 = NEPOOL ID (shifted left by 1), col 2 = something else. The NEPOOL value lands in col 1 which is unmapped, so `nepool_gis_id` is silently None for every row.
- **Repro detail:** Affects `.xlsx` files with merged header cells in the heuristic path. GMCS detection path is unaffected (it reads specific cells by coordinate). LLM path is unaffected (LLM reads the text and infers columns by value).
- **Recommended fix:** After column-keyword mapping, do a second pass to detect shifted-but-recognisable columns: if no nepool column was found but one column has all-numeric values matching 4–6 digits, infer it as nepool_gis_id. This is a heuristic, not a guarantee, but covers the common merge pattern.
- **Estimated effort:** M

### defect-import-05: Intra-file duplicate NEPOOL IDs not detected (**highest severity open**)
- **Test:** tests/test_import_stress.py::TestSheetStructureChaos::test_intrafile_duplicate_nepool_flagged (xfail)
- **Severity:** high
- **Repro:** Upload a file where two rows have the same NEPOOL GIS ID (e.g. 53984) but different array names ("Array Alpha" and "Array Beta"). The preview endpoint only checks NEPOOL against the DB (`nepool_map` built from existing Arrays). It does not check for duplicates within the uploaded file itself. Both rows appear in the preview with no collision warning.
- **Impact:** At commit time, `ingest_commit` upserts Arrays by `(tenant_id, name)`. Since the two rows have different names, two separate Array records are created. Both have `nepool_gis_id = "53984"`. The GMCS writer then generates two sheets for the same physical meter, splitting kWh attribution between them. Bill credits flow to the wrong clients. Bruce would not notice until the quarterly report shows impossible per-array generation numbers.
- **Recommended fix:** In `ingest_preview`, after `_normalize(rows)`, build a dict keyed by `nepool_gis_id` and flag any ID that appears more than once with a `provenance.intrafile_nepool_duplicate = True` flag on each affected row. Add an `intrafile_nepool_duplicate` entry to the `warnings` list. The UI already handles per-row flags (it already reads `provenance.nepool_collision`).
- **Estimated effort:** S

### defect-import-06: account_number > 40 chars passes through ingest without validation
- **Test:** tests/test_import_stress.py::TestDomainLogic::test_account_number_too_long_flagged_at_parse_time (xfail)
- **Severity:** low
- **Repro:** Send a row with `gmp_account_number` longer than 40 characters. `_normalize` passes it through unchanged. The preview returns it cleanly. When the operator commits, `ingest_commit` hits `UtilityAccount.account_number = String(40)` and SQLite silently truncates; PostgreSQL (Railway) raises `sqlalchemy.exc.DataError: value too long for type character varying(40)`, which surfaces as a 500 Internal Server Error to the operator with no indication of which account caused the problem.
- **Recommended fix:** In `_normalize`, after `clean_text`, check if `f == "gmp_account_number"` and `len(s) > 40`; if so, set `gmp_account_number = s[:40]` and add an `account_parse_warning` key, OR reject the value and add an `account_parse_error`. Truncation is safer than rejection since account numbers are not a uniqueness key in the DB.
- **Estimated effort:** S
